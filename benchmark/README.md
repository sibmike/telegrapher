# Compression Knowledge-Preservation Benchmark (CKPB)

A dual-suite multiple-choice QA benchmark designed to measure how well a text-compression method preserves the information a downstream task actually needs.

Most compression methods are eval'd by perplexity or compression ratio вЂ” but neither tells you whether the answer to a specific factual question survives. CKPB does: it scores compressed text on factual QA pairs generated from the original, with a **fine-facts** suite whose distractors are deliberately near-miss ("4.8%" vs "4.3%") so that only lossless methods can distinguish them.

The benchmark is **compression-agnostic**: plug in any `compress_fn(str) -> str` вЂ” extractive, abstractive, neural, rule-based вЂ” and get a single accuracy number and per-item results.

## Suites

| Suite | Pairs | Design intent |
|---|---|---|
| `key_facts` | 4,080 | Core concepts, headline findings, main claims. Distractors are generically plausible. Measures whether the *gist* survives. |
| `fine_facts` | 801 | Buried details: precise numerical qualifiers, conditional statements, boundary conditions. Distractors are near-miss variants. Measures whether *detail* survives. |

QA pairs were generated by GPT-4.1 from 4,081 LongBench-v2 chunks (max 1,000 words). See [`dataset_card.md`](dataset_card.md) for full provenance, known biases, and licensing.

## Quickstart

```bash
# from repo root
pip install -r code/requirements.txt
export OPENAI_API_KEY=sk-...

python -m benchmark.harness.example_run   # 10-item smoke test with identity baseline
```

To evaluate your own compressor:

```python
from benchmark.harness import run_benchmark

def my_compressor(chunk: str) -> str:
    # ... your logic ...
    return compressed

summary = run_benchmark(
    compress_fn=my_compressor,
    suite="fine_facts",
    mc_model="gpt-4.1-nano",
    output_path="my_results.csv",
)
print(summary)   # {"suite": "fine_facts", "n": 801, "accuracy": 0.87, ...}
```

## Data schema

Each line in `data/*.jsonl`:

```json
{
  "qid": "key-00000",
  "suite": "key_facts",
  "longbench_doc_id": "66eae4de5a08c7b9b35dd12d",
  "question": "...",
  "choices": ["...", "...", "...", "..."],
  "gold_idx": 3,
  "original_tokens": 1514,
  "generation_model": "gpt-4.1"
}
```

The benchmark intentionally does **not** redistribute LongBench-v2 chunk text. Fetch the source documents from [THUDM/LongBench-v2](https://huggingface.co/datasets/THUDM/LongBench-v2); the harness' chunk-lookup is matched by `original_tokens` within each document.

## Results schema

Each line in `results/*.csv`:

```
qid, suite, compressor, eval_model, gold_idx, predicted_idx, correct,
original_tokens, compressed_tokens, compression_ratio
```

## Current baselines

See [`results/leaderboard.md`](results/leaderboard.md) for the full table. Highlights:

- **Telegraph English** (~50% compression): 99.1% key_facts, 96.5% fine_facts (GPT-4o evaluator)
- **LLMLingua-2 @ 50%**: 99.0% key_facts, 93.3% fine_facts
- **LLMLingua-2 @ 33%**: accuracy drops sharply on fine_facts, especially with smaller eval models
